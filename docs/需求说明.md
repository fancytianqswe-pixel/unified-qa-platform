# 新任务与技能中心需求实现说明

## 本地服务启动与架构

**产品命名**：用户可见文案与文档中的产品名称为 **「统一质检平台」**；技术标识（如 Cookie **`xingyan_session`**、`xingyan_*` 的 `localStorage` 键、目录 **`xingyan_datasource_*`** 等）未改名，与「仅换展示文字」的约定一致。

1. **启动页面（http://localhost:3000）**  
   - 在仓库根目录进入前端工程：`frontend`  
   - 执行：`npm run dev`（脚本为 **`next dev --webpack`**，默认监听 **3000** 端口；避免 Next 16 默认 Turbopack 在含中文的长源码上触发 **`next-code-frame` UTF-8 切片 panic** 导致进程退出、`localhost` 连接被拒绝。）**勿**单独运行无 **`--webpack`** 的 **`next dev`**（易默认走 Turbopack）。若 **`http://localhost:3000/`** 仅显示**纯文本** **Internal Server Error**、无本站布局样式，多为 **开发进程已崩溃、3000 被其它程序占用、或 `.next` 缓存异常**：先看运行 **`next dev` 的终端**是否有 Node 栈；结束进程后执行 **`npm run dev:clean`**（删除 **`frontend/.next`** 再以 **`next dev --webpack`** 启动），并用系统工具确认 **3000** 端口由预期 Node 监听。根级未捕获错误由 **`src/app/global-error.tsx`** 提供独立 **`html/body`** 的中文降级页（**不依赖** **`I18nProvider`**）；若仍只见纯文本，说明响应可能**非本 Next 应用**（例如端口上为其它 HTTP 服务）。  
   - 浏览器访问：`http://localhost:3000`（登录页）；登录成功后进入工作台（默认跳转 **`/new-task`**）。同网段可用终端输出的 Network 地址访问。**静态资源**：品牌 Logo 固定放在 **`public/brand/logo.png`**（用户上传的平台图标；登录页 **`LoginPage`** 与新任务空状态 **`NewTaskChat`** 均用原生 **`img`** 引用 **`/brand/logo.png`**，**不**走 **`next/image`** 优化管道，避免个别环境下加载异常或显示为纯色块）。**`layout.tsx` 的 `metadata.icons`** 与可选 **`src/app/icon.png`** 同源。若 Logo 被误删或替换成 1×1 占位图，页面会显示色块而非品牌图——可从 Cursor 会话附件或备份恢复后覆盖该文件。**字体与图标依赖（国内友好）**：**`layout.tsx`** **不**使用 **`next/font/google`（Geist）**；正文字体栈为系统字体 + **PingFang SC / Microsoft YaHei**（见 **`globals.css`**）。界面图标使用 **`lucide-react` npm 包**（构建进 bundle，不依赖 Google CDN）；品牌图、站点图标一律 **本地 `public/`**，禁止在运行时从 **Google Fonts / gstatic / 国外图标 CDN** 拉取。  
2. **服务职责**  
   - **Next.js 开发服务器**同时提供：静态/SSR 页面、以及 `src/app/api/**` 下的 **Route Handlers**（如聊天、技能、任务、系统模型、数据源测试等）。  
   - 聊天、任务调度、技能注册等接口在服务端再 **转发** 到配置的下游（如 Hermes / OpenAI 兼容端点），需在环境变量中配置对应 URL；仅启动前端即可打开 UI，完整链路能力依赖下游与 `.env` 配置。  
3. **与「相关服务」的关系**  
   - 对 `localhost:3000` 而言，**必启**：`frontend` 的 `npm run dev`。  
   - **按需**：若需真实对话/任务/技能后端能力，需按项目文档启动或指向 Hermes 等后端，并设置 API 路由读取的环境变量。  
4. **「全部服务」在本仓库中的含义与启动顺序**  
   - **前端应用**：在 `frontend` 目录执行 `npm run dev`，得到 `http://localhost:3000`（Next.js + `app/api` 一体化）。  
   - **Hermes（可选）**：源码位于 `hermes-agent/hermes-agent-main`，官方提供 `docker-compose.yml`（`gateway` + `dashboard`，默认 `network_mode: host`）。本机需已安装并启动 Docker，在该目录首次建议执行 `docker compose up -d --build` 以本地构建镜像；构建过程需能访问 **`registry-1.docker.io`（Docker Hub）** 与 **`ghcr.io`** 拉取基础镜像，若超时/连接失败，需在可访问外网的环境、配置 **Docker 镜像加速/代理**，或在 Docker Desktop → Settings → Docker Engine 中为 `registry-mirrors` 等按需配置后再重试。  
   - **Docker Desktop（Windows / macOS）**：`network_mode: host` 绑定的是 **Docker 内部 Linux VM** 的回环地址，不是 Windows 上的 `127.0.0.1`；若 Dashboard 仅监听 `127.0.0.1:9119`，在宿主机浏览器访问会出现 **`ERR_CONNECTION_REFUSED`**。请在该目录改用专用编排文件（勿与默认 `docker-compose.yml` 同时叠加，避免 `network_mode` 冲突）：先 `docker compose down`，再执行 `docker compose -f docker-compose.docker-desktop.yml up -d --build`。该文件使用桥接网络 `hermes-net`，映射 **`8642→gateway`、`9119→dashboard`**，Dashboard 使用 **`--host 0.0.0.0 --insecure`**（Docker 端口转发需要监听非回环地址；`--insecure` 为上游 CLI 要求），并设置 **`GATEWAY_HEALTH_URL=http://gateway:8642`** 供容器内健康检查。完成后在浏览器访问 **`http://127.0.0.1:9119`**。Linux 本机仍可使用默认 `docker-compose.yml`。未成功拉起 Hermes 时，仅保留前端亦可浏览 UI，对话/任务等依赖 Hermes 的能力需另行部署或改指已有网关地址。  
   - **backend 目录**：当前为占位目录，无独立进程；业务 API 主要由前端的 Route Handlers 与下游服务承担。  
   - **数据库（MySQL，按需）**：数据源联调（`/api/datasource/test`、`preview`、`columns` 等）需要可连的 MySQL。测试实例说明见 **`docs/数据库测试数据.md`**（默认 **`127.0.0.1:3307`**，管理账号 **`root` / `Root#2026!AiCursor`**，库 **`qa_test_core.products`** 等见 **`docs/mysql_test_data.sql`**）。**对话草稿卡片**：`hermes-datasource` 中 **`password` 必须为真实口令**；若模型输出 **`***` 掩码**，`POST /api/datasource/test` 会按字面 `***` 连库并报 Access denied——草稿卡会提示在卡片内补填密码（**`isDatasourcePasswordPlaceholder`**），向导系统提示已禁止示例块使用 `***`。**草稿进度持久化**：助手消息上的 **`cards[].payload.draft`**（连通性结果、补填密码、字段勾选、样例行、是否已保存）随 **`sessionMessages`** 写入 **`chat-store-v1`**；**`chatStore.patchDatasourceDraftCard`** 在用户操作后回写，**`DatasourceDraftCard`** 重新进入会话时从 **`draft` 恢复**，不再每次挂载清空并重跑探测（指纹 **`datasourceDraftFingerprint`** 变更时丢弃旧 draft）。若未注册 Windows 服务 **`MySQL84Test`**，可在已存在 **`%USERPROFILE%\mysql-test.ini`** 与 **`%USERPROFILE%\mysql-data`** 的前提下，用 PowerShell 前台或后台启动：  
     `& "C:\Program Files\MySQL\MySQL Server 8.4\bin\mysqld.exe" --defaults-file="$env:USERPROFILE\mysql-test.ini"`  
     启动后确认 **`netstat -an | findstr ":3307"`** 出现 **`LISTENING`**。前端表单里连接测试库时请将端口改为 **3307**（与文档一致；UI 默认端口 3306 仅为占位）。
5. **本机一键核对（Windows）**  
   - **仅前端**：在 `frontend` 执行 **`npm run dev`**，终端出现 **`✓ Ready`** 且 **`Local: http://localhost:3000`** 后，先在 **`/`** 使用默认 **`admin` / `admin`** 登录，再访问 **`/new-task`** 等受保护路由；环境变量来自 **`frontend/.env.local`**（未配置 Hermes 时仍可浏览 UI，对话等能力依赖下游）。  
   - **Hermes**：先启动 **Docker Desktop**，直至 `docker ps` 不再报 **`npipe:////./pipe/dockerDesktopLinuxEngine`** 类错误；再在 **`hermes-agent/hermes-agent-main`** 执行 **`docker compose -f docker-compose.docker-desktop.yml up -d --build`**（与上文 §「Docker Desktop」一致）。  
   - **对话/技能 BFF 联调**：`HERMES_GATEWAY_URL=http://127.0.0.1:8642` 与 **`API_SERVER_KEY`** 需与 compose 中默认值或自定义值一致（见 **`frontend/.env.example`**）。

## 登录与访问控制（根路径 `/`）

1. **`http://localhost:3000/`** 为登录页：组件 **`src/components/auth/LoginPage.tsx`**，提交 **`POST /api/auth/login`**；服务端在 **`getAccessControlRuntimeState().users`** 中按 **`account`** 查找用户行，用 **`src/lib/password-auth.ts`** 的 **`verifyPasswordForUser`** 校验 **`passwordHash`**（PBKDF2-SHA256，盐由用户 **`id`** 派生，格式 **`v2$…`**）。**超级管理员**（**`u-super-admin`**）默认口令与 **`platform-auth.ts`** 中 **`SUPER_ADMIN_PASSWORD`** 一致（默认 **admin**，与账号 **admin** 相同）；**普通用户**在 **`mergeAccessControlInvariant`** 中若尚无哈希，则默认口令为 **`account`（与账号相同）**。若 **`id === u-super-admin`** 且 **`role`** 与 **`SUPER_ADMIN_ROLE_NAME`** 不一致则返回 **403**。校验通过后签发 **httpOnly** Cookie **`xingyan_session`**（载荷经 **Web Crypto HMAC-SHA256** 签名，密钥为 **`SESSION_SECRET`**，缺省为开发占位串，见 **`frontend/.env.example`**）。**`Secure` Cookie**：**`POST /api/auth/login`** 与 **`POST /api/auth/logout`** 通过 **`sessionCookieSecureFlag(request)`**（**`src/lib/session-cookie-secure.ts`**）设置 **`cookies.set`** 的 **`secure`**：仅 **`NODE_ENV === "production"`** 且为 **HTTPS**（含反向代理 **`x-forwarded-proto: https`**）时为真，避免 **`next start` + `http://`** 浏览器丢弃 **`Set-Cookie`** 导致「登录瞬断 / 无法登录」；开发模式下恒为 **false**。**`POST /api/auth/login`** 声明 **`export const runtime = "nodejs"`** 且业务体 **`try/catch`**，异常时返回 **500** JSON。已登录用户打开 `/` 时由前端 **`GET /api/auth/me`** 成功后 **`replace` 至 `/new-task`**。**`src/app/api/auth/me/route.ts`** 的 **`GET`** 整体 **`try/catch`**：读取 Cookie、验签、**`getAccessControlRuntimeState` / `buildSessionAccess`** 等任一步未预期异常时返回 **500** JSON（**`ok:false`**、提示稍后重试），避免抛错导致工作台壳层或 RSC 连带失败。**前端可见错误**：**`src/components/Providers.tsx`** 用 **`AppErrorBoundary`**（**`src/components/AppErrorBoundary.tsx`**）包裹 **`I18nProvider`**，子树内未捕获的 **React 渲染错误**以全屏卡片展示 **message / digest / stack / componentStack**；**`ClientRuntimeDiagnostics`**（**`src/components/ClientRuntimeDiagnostics.tsx`**）监听 **`window.error`** 与 **`unhandledrejection`**，在页面底部琥珀色条展示最近若干条客户端异常（可收起/清空）。**`LoginPage`** 对 **`GET /api/auth/me`**：**401 / 403**（如 **`{"ok":false,"message":"未登录"}`**）视为**未登录正常态**，**不**写入 **`sessionProbeError`**；其余非 **2xx**、**2xx** 但正文非合法 JSON、或 JSON **`ok !== true`** 时，将 **HTTP 状态与响应正文截断** 写入 **`sessionProbeError`**，在「检查登录态」卡片与登录表单上方 **`pre`** 展示；**`POST /api/auth/login`** 失败时在 **`login.fail`** 文案下附加响应 **`raw`** 截断，**`catch`** 时附加 **`Error.name/message`**，便于浏览器内截图排查而不仅依赖终端。**登录页语言**：组件内 **`LoginLanguageSelect`** 在登录表单下方（及「检查登录态」卡片底部）提供语言下拉，选项与 **系统管理 → 常规** 一致（**`ui-preferences.ts`** 的 **`APP_LOCALES` / `LOCALE_LABELS`**：**简体中文、繁體中文、English、日本語、한국어、Français、Deutsch**），经 **`useI18n().setLocale`** 写入 **`localStorage`** 键 **`xingyan_ui_locale`**，未登录即可切换界面文案，登录后工作台沿用同一偏好。
2. **`src/middleware.ts`**：对 **`/new-task`、`/skills-center`、`/task-center`、`/data-center`、`/conversation`、`/system-settings`、`/api/*`** 等（matcher 列出）校验会话；**`normalizePathname`** 去掉非根路径的尾部 **`/`**，使 **`/api/auth/login/`**、**`/api/auth/logout/`** 与无斜杠形式一致地命中放行白名单，避免 **POST 登录被误判为需会话而返回 401**；**`GET /api/auth/me`** 在 middleware 中 **`NextResponse.next()`** 放行，由 **Node** 上的 **`api/auth/me/route.ts`** 统一验签（**`export const runtime = "nodejs"`**），避免在 **Edge middleware** 与 Route Handler 双路径验签时出现 **Web Crypto / Cookie** 行为差异导致首屏会话探测失败。未登录访问页面时 **302** 至 **`/?from=<原路径>`**；未登录访问受保护 API 时返回 **401 JSON**（**`/api/auth/login`、`/api/auth/logout`、`/unauthorized`** 放行）。**`middleware` 主体 `try/catch`**：验签或重定向等未捕获异常时，对 **`/api/*`** 返回 **503** JSON（**`ok:false`**），对页面请求 **302** 至 **`/`**（清空 query），并 **`console.error('[middleware]', …)`**，减轻 Edge 异常表现为整页 **Internal Server Error**。**例外**：**`/api/datasource/*`** 仍放行，以便 Hermes 网关在 Docker 内以 HTTP 调用数据源 BFF（无浏览器 Cookie；仅限内网联调场景，公网部署应再加网关鉴权或 mTLS）。**细粒度路由**：菜单/系统管理入口由 **`GET /api/auth/me`** 返回的 **`access`**（**`session-access.ts`** 基于 **`getAccessControlRuntimeState`** 与用户 **`role` 名称**匹配角色行）驱动；**`DashboardShell`** 过滤主导航、系统弹窗子项，并在无权限访问当前路径时 **`replace`** 到首个允许页；**`/system-settings`** 无任一 **`sys-*`** 权限时退回工作台。配置了 **`HERMES_ACCESS_CONTROL_ENDPOINT`** 时，**`/api/system/access-control`** 在 **GET/POST** 从上游 Hermes 拿到合法 **`{ orgTree, users, roles }`** 后须 **`persistAccessControlPayload`** 同步进 **`getAccessControlRuntimeState`**，否则 **`/api/auth/me`** 仍读本地旧态，会出现「角色页已保存系统权限、侧栏普通用户仍无系统管理」的不一致。
3. **浏览器端多账号数据隔离**：**`client-data-scope.ts`** 将当前用户 **`sub`** 写入 **`localStorage`**（**`xingyan_client_data_scope_sub`**）；**`chat-store-v1`** 的 Zustand **`persist`**、**`datacenter.datasources.v1`**、**`skills-center.*`** 及 **`skills-api-cache`** 的读写键均带 **`::user::<sub>`** 分桶。登录成功（**`LoginPage`**）与 **`/api/auth/me`** 发现 **`sub`** 变更时调用 **`rehydrateChatStoreForDataScope`** 并失效技能 API 缓存，避免 **wenbin** 看到 **admin** 的会话历史与本地数据中心列表。退出登录时清除分桶指针并重载 **guest** 分桶。**升级前**全局键：聊天 **`chat-store-v1`**、数据源 **`datacenter.datasources.v1`**（无 **`::user::`**）。对超级管理员（**`sub === u-super-admin`**）：**`chatStore`** 的 **`persist.getItem`** 解析新旧两键的持久化 JSON，按 **`historySessions` 条数 + `sessionMessages` 键数 + `modelConfigs` 条数** 比较「数据丰富度」；若旧键 **`chat-store-v1`** 更富，则写入分桶键并删除旧键（避免分桶里已落过「空状态」后无法再读回升级前数据；**模型配置与历史会话同包**，一并恢复）。**`datasource-storage.ts`** 的 **`loadDataSourcesFromStorage`** 在同样身份下，若未分桶键里数据源**数组条数**大于分桶内数组，则将旧键内容迁到分桶键、删除旧键并派发 **`datacenter-datasources-changed`**。其它账号的旧全局键不自动迁入，以免把混用期的数据误绑到普通用户。若用户已清空相关 **`localStorage`** 键，则无法从浏览器本地恢复。
4. **与「系统管理 → 用户与权限」打通**：**`UserPermissionSection`** 挂载后 **`GET /api/system/access-control`** 拉取 **`accessControlRuntime`** 中的组织/用户/角色（**`sanitizeAccessControlForClient`** 去掉 **`passwordHash`**，不下发哈希）；本地编辑后 **约 900ms debounce** **`POST`** 同一接口写回；角色页 **「保存」** 会 **`await`** 立即 **`POST`** 一次，避免用户只点保存未等满 debounce 时误以为未落库。**只读能力**：前端用 **`useSessionAccess()`** 与 **`canUsePageButton(access, pageKey, 按钮文案)`**（**`session-access.ts`**）对照 **`/api/auth/me`** 下发的 **`access.menuKeys` / `access.buttonPermissions`**。叶子页键 **`sys-user-permission-user`** 下「新增组织 / 删除组织 / 新增用户 / 删除用户」分别控制组织树与用户表操作；**`sys-user-permission-role`** 下「新增角色 / 保存 / 配置按钮」分别控制新增角色、写库与菜单矩阵勾选。已勾选某叶子菜单但**该叶下按钮权限全不选**时，仍可进入对应页签**浏览**列表与矩阵，控件 **`disabled`**，并在界面附简短只读说明。新建用户可在表单填 **`__plainPassword`**（前端仅新增行带该字段；留空则与账号相同），服务端 **`access-control/route.ts`** 用 **`hashPasswordForUser(plain, id)`** 写入 **`passwordHash`**；未改密用户从**当前内存态**合并保留原哈希；**禁止**请求体携带客户端 **`passwordHash`**。写库成功后响应同样经 **`sanitizeAccessControlForClient`**，前端用 **`data`** 覆盖 **`orgTree` / `users` / `roles`**，避免明文密码字段长期留在浏览器状态。服务端 **`mergeAccessControlInvariant`**（**`accessControlRuntime.ts`**）在每次 **GET 归并** 与 **POST 写入** 时强制：存在 **`u-super-admin`** 且 **`account === admin`**、**`role === 超级管理员`**；若缺少 **`role-normal`（普通用户）** 且非单租户模式则**自动补回**内置权限（新任务/任务中心及对应按钮）；对仍缺 **`passwordHash`** 的用户行补默认哈希（超级管理员用 **`SUPER_ADMIN_PASSWORD`**，其余用 **`account`**）。**`SINGLE_SUPER_ADMIN_DEPLOYMENT`**（**`platform-auth.ts`**）默认 **`false`**：可新增用户/角色；为 **`true`** 时仅保留超级管理员用户与 **`role-super`**，并隐藏「新增用户 / 新增角色」。
5. **工作台**：**`DashboardShell`** 侧栏用户信息区主标题为 **`account`**，副标题为 **`role`**，头像由 **`getUserAvatarGlyph(name || account)`**；信息块为 **`rounded-2xl bg-white p-2`**，**无边框**；**不再**在侧栏头像悬停层提供 **退出登录**（改在 **系统管理 → 常规** 内显式按钮）。**收起 / 展开**与主导航、历史、系统入口为 **`size-9` 点击格 + `size-5` Lucide 图标**；收起态历史灰底容器为 **`flex-none`**（**`slice(0, 7)`**），展开态历史区 **`flex-1`**。全局壳层与登录/无权限等入口文案经 **`I18nProvider`**（**`src/i18n/`**）与 **`useI18n().t`**，支持 **zh-CN / zh-TW / en / ja / ko / fr / de**，语言持久化 **`xingyan_ui_locale`**；主题仅 **浅色 / 深色**，持久化 **`xingyan_ui_theme`**，根节点 **`html`** 加 **`dark`** 类配合 **`globals.css`** 中 **`@custom-variant dark`**，**不**跟随系统深浅色。

### 国际化：表层 + 深层文案

- **词条分层（固定标准）**：每种语言由两层组成——**`src/i18n/messages/<locale>.ts`**（导航、登录、系统弹窗侧栏、通用等「表层」）+ **`src/i18n/messages/deep-<locale>.ts`**（系统管理内页、模型、MCP、数据中心、技能中心、聊天内置技能等「深层」）。**`deep-*` 的键集合须与 `deep-en.ts` 完全一致**（复制 `deep-en` 后只改文案值）；规范说明见 **`src/i18n/deep-locale-contract.ts`**，合并表在 **`registry.ts`** 的 **`bundles`**。**当前七种语言均各有一份深层包并已接入**：**`zh-CN`** → **`deepMessagesZhCN`**（`deep-zh-cn.ts`）；**`zh-TW`** → **`deepMessagesZhTW`**（`deep-zh-tw.ts`）；**`en`** → **`deepMessagesEn`**；**`ja`** → **`deepMessagesJa`**；**`ko`** → **`deepMessagesKo`**；**`fr`** → **`deepMessagesFr`**；**`de`** → **`deepMessagesDe`**。新增语言时复制 **`deep-en.ts`** 为 **`deep-<locale>.ts`**，在 **`registry.ts`** 与 **`ui-preferences.ts`** 注册后，可用脚本对比 **`deep-en`** 与各 **`deep-*`** 的键集（须 **missing=0、extra=0**）。缺键时 **`I18nProvider`** 仍按 **当前语言 → en → zh-CN** 回退。

## Hermes `docker-compose.docker-desktop.yml` 逻辑说明

1. **问题**：默认 `docker-compose.yml` 使用 `network_mode: host`，在 Docker Desktop 上「主机」为内部 VM；Dashboard 命令为 `--host 127.0.0.1` 时，**9119 仅在 VM 回环可访问**，Windows 浏览器访问 `127.0.0.1:9119` 会被拒绝连接。  
2. **做法**：单独提供 `docker-compose.docker-desktop.yml`（**完整替代**该场景下的 compose，不与默认文件 `-f` 叠加），为 `gateway` / `dashboard` 加入 **`hermes-net` 桥接网络**，声明 **`ports: "8642:8642"`、`9119:9119`**，将 Dashboard 改为 **`--host 0.0.0.0`** 且必须加 **`--insecure`**（上游对非 localhost 绑定的校验，否则进程退出、映射端口仍拒绝连接），并设置 **`GATEWAY_HEALTH_URL=http://gateway:8642`**，使 Dashboard 容器通过 Compose 服务名访问 Gateway。  
3. **数据与镜像**：与官方一致使用 `~/.hermes` 卷、`hermes-agent` 镜像；`gateway` 仍 `build: .`，`dashboard` 复用同一镜像。  
4. **Gateway 状态为 STOPPED**：Dashboard 用 `GATEWAY_HEALTH_URL` 请求 Gateway 的 **`/health`**，该路由由 **`API_SERVER` 平台**（`gateway/platforms/api_server.py`）提供；未启用 API Server 时探测失败。`docker-compose.docker-desktop.yml` 中为 `gateway` 设置 **`API_SERVER_ENABLED=true`、`API_SERVER_HOST=0.0.0.0`、`API_SERVER_PORT=8642`**；绑定 `0.0.0.0` 时上游要求 **`API_SERVER_KEY`**（可用环境变量覆盖 compose 内默认值）。调用 OpenAI 兼容接口时需携带该 Key。  
5. **统一质检平台数据源工具（容器内 Gateway → 宿主机 Next）**：`docker-compose.docker-desktop.yml` 的 `gateway` 已设 **`DATASOURCE_MCP_BASE_URL`** 默认 **`http://host.docker.internal:3000`**（可用同名环境变量覆盖），供 `tools/xingyan_datasource_tools.py` HTTP 调用 Next **`/api/datasource/*`**。Linux 使用默认 **`docker-compose.yml`**（`network_mode: host`）时默认 **`http://127.0.0.1:3000`**。  
6. **API Server 对话「MCP 优先于 skill_view」**：`gateway/platforms/api_server.py` 在创建 **`AIAgent`** 时，若 **`HERMES_XINGYAN_PREFER_MCP_TOOLS`** 非 `0/false/off`（compose 默认 **`1`**），将一段简短策略拼入 **`ephemeral_system_prompt` 前缀**，引导模型对文档解析等任务**先**用已注册 MCP（如 **`mcp_*_mineru_*`**），**再**用 **`skill_view`** 加载技能（如 `ocr-and-documents`）；关闭行为：`HERMES_XINGYAN_PREFER_MCP_TOOLS=0`。

7. **Hermes 容器内 MinerU MCP（`xingyan-mineru-api`）长期「用不了」的根因归类**（与 `tools/mcp_tool.py`、`docker-compose.docker-desktop.yml` 行为一致）：
   - **HTTP 根地址不可达（最常见）**：`mineru-api-mcp` 仅通过 **`MINERU_API_BASE_URL`** 访问 FastAPI。若 **`mineru-api` 实际监听在 WSL** 而 Hermes 在 **Docker Desktop Linux VM** 内，则 **`http://host.docker.internal:8000`** 往往指向 **Windows 侧无监听**，子进程内 `fetch` 会得到 **ECONNREFUSED**，工具表现为健康检查失败或解析失败。对策：与 Gateway **同一 Compose 网络** 起 sidecar（`http://mineru-api:8000`，见 `docker-compose.mineru-api.sidecar.yml` 与 `docs/MinerUDockerSidecar.md`），或改为 Windows/容器均可路由到的 API 地址。
   - **配置未按「运行 Gateway 的环境」书写**：容器内 **`HERMES_HOME=/opt/data`**（卷挂载宿主 **`~/.hermes`**），`mcp_servers` 从该 **`config.yaml`** 读取；**`args` 中 `dist/index.js` 路径**须为容器内存在路径（Compose 已将仓库 **`mcp-servers` → `/opt/xingyan-mcp`**），且宿主需已 **`npm run build`** 出 **`dist`**。路径错误或缺 **`dist`** 会导致 **`FileNotFoundError`**，网关日志出现 **`Failed to connect to MCP server`**。
   - **`platform_toolsets.api_server` 的 MCP 白名单效应**：`hermes_cli/tools_config.py` 中 `_get_platform_tools` 规定：若 **`api_server`** 的工具集列表里**显式写出了部分 MCP 服务器名**，则**只启用这些名字**，未写入的已配置 MCP（含 **`xingyan-mineru-api`**）不会进入工具集，模型侧表现为「没有 MinerU 工具」。未显式写任何 MCP 名时，才会默认并入**全部**已启用 **`mcp_servers`**。
   - **工具名与文档示例不一致**：Hermes 注册名由 **`sanitize_mcp_name_component`** 将 **`xingyan-mineru-api`** 中的连字符等转为下划线，实际前缀为 **`mcp_xingyan_mineru_api_`**（例如 **`mcp_xingyan_mineru_api_mineru_api_health`**），而非字面量带连字符的片段。
   - **解析阶段读盘路径**：`mineru-api-mcp` 在 **Gateway 进程所在机器** 上 **`readFile(filePath)`** 再上传；`filePath` 必须是 **容器内可读绝对路径**（如 **`/opt/data/chat-uploads/...`**），不能指望 MCP 读 Windows 盘符或仅 WSL 内路径。
   - **可选前缀限制**：若配置了 **`MINERU_API_ALLOWED_READ_PREFIX`**，`filePath` 必须落在该前缀下，否则 MCP 在本地即拒绝。
   - **观测位置**：stdio MCP 的 **stderr** 写入 **`~/.hermes/logs/mcp-stderr.log`**（与网关日志中的 **`Failed to connect to MCP server`** 并列可查）；Python 侧若未安装 **`mcp`** 包会跳过发现（官方 Docker 镜像通过 **`uv pip install -e ".[all]"`** 已包含 **`hermes-agent[mcp]`**，一般不是根因）。

### 登录与权限数据（多账号口令）

- **`access-control-constants.ts`**：仅导出 **`ROOT_PLATFORM_ORG_ID`**、**`NORMAL_USER_ROLE_ID`** 等无 Node 依赖常量。**`UserPermissionSection`（客户端）**须从此文件引用组织根 id，**禁止**从 **`accessControlRuntime.ts`** 引用，否则 Webpack 会把内含 **`crypto`** 的 **`password-auth`** 打进浏览器包并触发 **`node:crypto` / UnhandledSchemeError**。服务端 **`accessControlRuntime.ts`** 仍再导出上述常量以保持 **`import { … } from "@/lib/accessControlRuntime"`** 的兼容。  
- **`password-auth.ts`**：`hashPasswordForUser` / `verifyPasswordForUser`，PBKDF2-SHA256，迭代次数与 **`userId`** 编码写入 **`v2$…`** 串；盐由 **`sha256("xingyan-pbkdf2|" + userId)`** 派生，避免每次 **`mergeAccessControlInvariant`** 补哈希时随机盐导致内存态 JSON 比较抖动。  
- **`accessControlRuntime.ts`**：**`UserRow.passwordHash`** 仅存服务端；**`mergeAccessControlInvariant`** 末尾对缺哈希用户：超级管理员用 **`SUPER_ADMIN_PASSWORD`**，其余用 **`account`** 作为默认明文再哈希。**`sanitizeAccessControlForClient`** 供 API 响应剥离 **`passwordHash`**。  
- **`/api/system/access-control`**：**`GET`** 本地路径返回脱敏数据；**`POST`** 用 **`mergeUsersWithPasswordPolicy`**：若行带 **`__plainPassword`** 则重算哈希；否则沿用 **`getAccessControlRuntimeState`** 中同 **`id`** 的旧哈希；校验拒绝请求体中的 **`passwordHash`**。  
- **`/api/auth/login`**：按 **`account`** 命中用户行后 **`verifyPasswordForUser`**，通过则 **`signSessionPayload`**（**`sub`/`account`/`name`/`role`** 与行一致）。  
- **`UserPermissionSection`**：新增用户时可选密码框，写入行的 **`__plainPassword`**；debounce **POST** 成功后以响应 **`data`** 覆盖状态，去掉明文字段。

## 代码实现逻辑梳理

1. 用户在中央输入框输入文本或语音，底部三个快捷药丸（**`QUICK_CHIP_IDS`**）：展示名 **`getQuickChipSkillLabel`**、预填 **`getQuickChipPrompt`**（均来自 **`quick-chip-builtin.ts`** → 技能中心 **`skill.builtin.*`** / **`builtin-*.ts`**，与详情页 **「一键试用」** 的 `samplePrompt` 同源）。逻辑封装为 **`applyQuickChipPreset`**（后台拉列表并 **`syncQuickChipSkillCardFromCatalog`** 对齐 Hermes `skillId`），与 URL **`workspacePreset`** 共用；点数据规则审核时 **`setDataRuleAuditWizardActive(true)`**。  
2. 输入提交后，`chatStore` 先插入用户消息，再按输入意图生成 AI 消息与卡片数据。  
3. 消息列表按时间倒序渲染，新消息显示在上方，并在顶部展示“最近一次追问上下文”提示。  
4. `AdaptiveCardRenderer` 根据 AI 返回的 `type` 动态分发卡片组件：  
   - `data_preview` -> `DataPreviewCard`  
   - `execution_plan` -> `ExecutionPlanCard`  
   - `error_diagnosis` -> `ErrorDiagnosisCard`  
   - `skill_confirm` -> `SkillConfirmCard`  
5. `DataPreviewCard` 展示前 5 条样例、字段映射与媒体缩略图，映射列表支持拖拽重排。  
6. `SkillConfirmCard` 点击确认后调用 `/api/skills/register`，成功后更新按钮状态。  
7. 整体样式采用大圆角、留白、极简布局，贴近钉钉悟空视觉风格。  
8. **过程区胶囊与 Hermes 信封 JSON**：折叠条摘要由 **`frontend/src/lib/process-step-heuristic.ts`** 的 **`summarizeHermesContentPartEnvelope`** 等启发式生成。上游常把一步结果包成 **`[{"type":"input_text","text":"…"}]`**，其中 **`text`** 可能是**再嵌一层 JSON**（例如技能就绪元数据 **`success`、`name`、`description`**）。**`tryParseJsonArray`** 会从整段字符串里**定位首个 `[` 再截取**后解析，避免行首有 **`>_` 等前缀**时 `JSON.parse` 整段失败而退回原始长串。解析出内层对象后：若 **`success === true`** 且存在 **`name`**，胶囊写**载入技能「name」及说明**（意图句，不写成败）；若 **`success === false`** 则写**请求加载技能「name」配置**（不把 **`error`** 原文塞进胶囊）。当外层不是数组而是**单个 `{ type, text }` 对象**时，先规范为数组再走同一套逻辑。若 **`JSON.parse` 仍失败**，先尝试 **`tryLooseHermesShellSuccessBlob`**：在仍像 Hermes **`input_text` 信封**的原文中用正则匹配 **`status: success`** 与 **`output`** 路径（含 SSE 截断、整段 **`[{`** 无法 `parse` 的情形），产出「拉取会话或解析流程产出的 JSON 结果文件」等**同步意图句**，避免胶囊退化为整段 JSON；**`tryLooseHermesEnvelopeHeuristic`** 在 **`JSON.parse` 仍失败**时识别 **`status":"error"`** 与 **`output`（Traceback/ModuleNotFoundError）**、**`httpStatus`+`202`/`task_id`/pending**、**仅 `output` 的目录列表** 等；**`tryLooseHttpApiInHermesBlob`** 对 **`httpStatus`+任意三位状态码**（含 **202**）与 **`task_id`/`file_names`** 归纳异步解析类请求；**`summarizeToolInnerWithResultPeeling`** 沿 **`result`/`data`/`payload`/`body`/`text`** 字符串多次剥壳；再用 **`tryLooseSkillMetaSummary`** 在原文中匹配 **`success: true`** 与 **`name` / `description`**；除未转义的 **`"name"`** 外，同时兼容 **`\\"name\\"`** 等**转义引号形态**（`stepLogs.message` 常为「一整段未再解析的字符串」，内层 JSON 键名在文本里仍以反斜杠加引号出现，旧版只认 **`"name":`** 会漏匹配）。**`type` 字段**若带首尾空格或大小写差异（如 **`INPUT_TEXT`**），经 **`normHermesPartType`** 归一后再识别 **`input_text` / `output_text`**，避免误判为非信封而直接预览 JSON。**`text` 内层**常见 **`{"result":"{\\\"httpStatus\\\":200,...}"}`**（业务 HTTP 结果再 JSON.stringify 一层），旧逻辑只在**顶层**找 **`httpStatus`** 会失败；现由 **`summarizeToolInnerWithResultPeeling`** 沿 **`result` 字符串**多次解析后再走 **`summarizeToolJsonInner`**，并在 **`JSON.parse` 失败或 SSE 截断**时由 **`tryLooseHttpApiInHermesBlob`** 在 Hermes 信封或 **`{` 载荷**上匹配 **`httpStatus`/`rows`** 归纳为一行意图（与仅匹配 **`status:success`** 的 **`tryLooseHermesShellSuccessBlob`** 互补）。正文含弯引号（U+201C 等）时先经 **`normalizeJsonLikeAsciiQuotes`** 再 `JSON.parse`。**`prefersLlmStepSummary`**：启发式已产出**意图型**稳定中文（**`looksLikeToolIntentPillLine`** 前缀表，含 **`载入技能「…」`** 等）时**不再**调用 **`POST /api/chat/process-steps-summarize`**，防止模型把 JSON 复读回来覆盖胶囊；**`MessageList`** 若已缓存的 LLM 一行仍像 **`[{ "type":…`** 则**丢弃**并回退启发式。终端类字段（**`status`/`command`/`exit_code`**）及 **`summarizeToolJsonInner`** 覆盖的常见 Hermes 内层结构（**`status` 为 success 且仅有 `output` 路径**、仅有 **`output`+`exit_code`**、**`bytes_written`** 写文件结果、**`success`+`diff`** 补丁摘要、端口探测/CONNECTION OK/Traceback 等模式）走专门一句中文；**`POST /api/chat/process-steps-summarize`** 仍为**回合结束后的异步补全**，不阻塞上游工具调用；胶囊主文案以**同步启发式**为先，避免「回合已结束但模型摘要未到」时长期裸露 JSON。其余才落到「工具返回：」前若干字符或仅在启发式仍像生 JSON 时再走模型润色（见 **`docs/LLM对话UI规范.md`**）。

## 技能中心代码实现逻辑梳理

**Hermes 技能底层（Gateway 原生 API + BFF）**：技能中心由 Next **`/api/skills/*`** 在服务端 `fetch` Hermes **Gateway API Server**（与对话 **`HERMES_TURN_ENDPOINT`** 同源主机，默认端口 **8642**）。本仓库已在 **`hermes-agent/hermes-agent-main/gateway/platforms/skills_api.py`** 实现与 **`api_server.py`** 注册的三个端点，直接扫描 Hermes 的 **`~/.hermes/skills`** 与 **`skills.external_dirs`** 下的 **`SKILL.md`**，与 Agent 侧 **`agent.skill_utils`** 一致。

| Hermes 端点 | 方法 | 响应约定 |
| --- | --- | --- |
| `/api/skills` | GET | `{ ok, list }`，`list` 元素为仪表盘 **`Skill`** 形状（`id` 为 **`h{根索引}:{相对目录}`**，如 `h0:my-skill`） |
| `/api/skills/detail` | GET | `?id=` → `{ ok, skill }`，含完整 **`skillDocMarkdown`**；当 **`id` 为 Hermes 形 `h{n}:datasource-wizard-skill` 等** 而 Hermes 无记录时，经 **`findBuiltinMockFallbackForDetailId`** 映射到内置 **`datasource-wizard-skill` / `skill-creator-skill`** 再 **`enrichSkillWithBundledMarkdown`** 兜底 |
| `/api/skills/register` | POST | 体同注册 BFF → `{ ok, skillId, version, message }`，在 **`~/.hermes/skills/<slug>/`** 落地 **`SKILL.md`**（调用 **`skill_manager_tool._create_skill`**） |

鉴权与 **`/v1/chat/completions`** 一致：请求头 **`Authorization: Bearer <API_SERVER_KEY>`**（或部署使用的 **`HERMES_API_KEY`**）。

前端侧逻辑在 **`src/lib/hermes-skills-client.ts`** 与 **`src/app/api/chat/turn/route.ts`**：显式配置了 **`HERMES_GATEWAY_URL`** / **`HERMES_TURN_ENDPOINT`** 时优先使用；**未配置时仍会依次尝试** `http://127.0.0.1:8642`、`http://localhost:8642`、`http://host.docker.internal:8642` 等默认网关根（便于本机与「Next 在 Docker、Gateway 在宿主机」联调）。另可与显式 **`HERMES_SKILLS_*_ENDPOINT`** 合并候选 URL。**连远端都不可达**时，技能列表/详情回退本地 **`skillBuiltinList` + `skillMockList`**；注册接口可有本地兜底说明。

| 环境变量 | 用途 |
| --- | --- |
| `HERMES_GATEWAY_URL` | **推荐**，网关根 URL（如 `http://127.0.0.1:8642`），用于 **`/api/skills*`** 与**对话**流式探测 **`/v1`** |
| `HERMES_TURN_ENDPOINT` | 可选；对话已配时可推导网关根（从 `/v1/...` 去掉 **`/v1`**）；不配时仍可用 `HERMES_GATEWAY_URL` 与默认候选 |
| `HERMES_SKILLS_LIST_ENDPOINT` / `HERMES_SKILLS_DETAIL_ENDPOINT` / `HERMES_SKILLS_REGISTER_ENDPOINT` | 覆盖默认路径的完整 URL（兼容旧部署或反向代理前缀） |
| `API_SERVER_KEY` / `HERMES_API_KEY` | 访问 Gateway 时携带 **Bearer** |

更完整的平台级说明见 **`docs/Hermes集成需求说明.md`**（§14.2、§16.3）。

**技能列表/详情性能与路由（BFF + 前端）**：`hermes-skills-client` 对多个 Gateway 根的 **`GET /api/skills` / `GET /api/skills/detail`** 由**顺序长超时累加**改为 **`Promise.any` 并行**、单端点约 **4s** 超时，墙钟时间近似「最快成功的一路」而非多地址相加。`GET /api/skills/list` 带 **`Cache-Control: public, max-age=12, stale-while-revalidate=120`**，减轻反复进入技能中心的等待。详情 **`/api/skills/detail`**、**`skills-center/[id]`** 与 **`SkillDetailView`** 使用 **`normalizeSkillDetailRouteId`**（安全多次 `decodeURIComponent`）；**`SkillCard`** 的 **`href`** 对 **`skill.id`** 使用 **`encodeURIComponent`**，避免 Hermes 形 **`h{n}:slug`** 在路径中与编码不一致导致详情 id 对不齐。**`SkillDetailView`** 在 **`fetch` 异常、`!res.ok` 或响应非 JSON** 时仍走 **`findBuiltinMockFallbackForDetailId` + `loadLocalSkills`**，避免「列表有技能、详情却未找到」。**`/api/skills/detail`** 外层 **`try/catch`**：服务端异常时若有内置兜底则仍返回 **200** 与 **`skill`**。

- **产品流程（由模型执行）**：依次确认 **数据来源**、**审核规则**、**输出结构**。规则与输出：**用户给出几条规则就只输出几条逐规则结论**，**禁止**在用户规则外擅自扩充（如自编规则⑤、金额/经济事项/考核等）除非用户明确要求；用户已逐条限定时，输出骨架为 **极短基本信息 + 整体结论 + 仅对用户 N 条规则** 的通过/不通过及原因。**执行节奏（渐进式）**：须按 SKILL 与向导提示 **分步推进**（数据源 → 规则与输出 → 阶段 A 单条 → 附件与 MinerU → 结论），**每步结束自检**后再进入下一步；**禁止**信息未齐时并行「全量取数 + 全附件解析 + 长报告」；**新会话/新任务**不得默认沿用上一对话的报账单号、路径、数据源，除非用户在本轮正文写明。**阶段 A**：**严格 1 条**；用户未指定行时模型须**自主从数据源选定 1 条**并一句话说明依据，再跑通取数→**有附件则先经 MinerU（MCP）解析**→**仅按用户规则**判定→按约定结构输出；**阶段 A 结束后须询问用户确认**。**阶段 B**（v2.1 强化）：**仅**在用户确认 A 后，**逐条**审至多 5 单（非「5 条并行」）；须先 **DB 样本与 MinIO 目录对齐**（禁止默认 preview 前 5 行即等于可审单）；每单 **TYA… → BFF `/opt/data/chat-uploads/…` → MinerU**，**禁止**容器内 Python/minio SDK 直连 `192.168.*`；**MinerU** 仅用 **`parse_and_wait`**，**`backend=pipeline`**，**同一轮助手回复最多 2 次**调用，大 PDF 单独排队，失败则停损勿批量叠 10+ 次。本版**不**注入数据源 function tools（与数据源向导区分）。**附件正文**须在 SKILL 与 **`DATA_RULE_AUDIT_WIZARD_SYSTEM_PROMPT`** 中声明：**必须先 MinerU**，禁止未解析即编造；无 MinerU 时引导开启或请用户粘贴解析结果。**`api_server.py`** MCP 优先提示与阶段 B 纪律一致。  
- **报账单附件**：库表以 **`附件存储类型` / `附件存储桶` / `附件对象前缀`** 为对象存储权威；设置 **`MINIO_BUCKET`** 时 **`import_qa_test_baodan.mjs`** 写入 minio、桶与 `{MINIO_OBJECT_ROOT}/<报账单号>/`（与 sync 脚本对象键一致）。**MinIO 端点与 AK/SK 不入库**，由 Next/BFF **`MINIO_*`** 持有；技能**禁止向最终用户索要 Endpoint/密钥**。附件清单由 **桶 + 前缀 + BFF ListObjects** 得到；表列 **`本地附件目录`**、**`附件索引`** 已移除（存量删列见 **`docs/baodan_reimbursement_drop_legacy_attach_columns.sql`**）。技能须**先解析三列**，有宣告则走 MinIO → **`chat-uploads` 注入**；**禁止**用 preview 中 Windows 路径在容器内读盘。刷新三列可重跑 **`import_qa_test_baodan.mjs`**（可 `SKIP_SCHEMA=1`）或 **`patch_baodan_minio_columns.mjs`**。磁盘树推桶见 **`scripts/sync_baodan_to_minio.mjs`**。**主数据**：`<报账单号>.json` 仅作 **`import_qa_test_baodan.mjs`** 灌库来源；若字段已落在 **`reimbursement_bills` 各列**（与 `qa_test_baodan_schema.sql` 一致），可删除该 json，**审核以库表列为准**，MinerU 只处理 PDF/影像等附件正文；乱码文件名须 MinerU 按正文归类。**联调**：调用 **`/api/chat/turn`** 时可在 **`context.baodanStageBillNo`** 显式传入报账单号触发 BFF 拉取（`baodan-hermes-staging.ts` / `minio-baodan-staging.ts`）。**BFF 行为**：在数据规则审核向导下，`context.baodanStageBillNo` **支持逗号分隔多单**；除 `context` 与本轮 `text` 外，会从 **`conversationHistory` 由新到旧**在用户/助手消息中解析 **`TYA…`**（至多 **`BAODAN_STAGE_MAX_BILLS`** 个，默认 5），**每单独立**写入 `chat-uploads/<uuid>/`，避免阶段 B 复用阶段 A 的 uuid 去读另一单的相对路径。**仅配置 MinIO 而未传该字段时，历史上 BFF 不会拉对象**——须传 `baodanStageBillNo` 或在统一质检平台对话正文中写出 **`TYA…` 报账单号**（由 `chatStore` 与 BFF 按 **`resolveBaodanBillNosFromConversation` / `resolveBaodanBillNosForStaging`** 合并历史并写入 `context`）。**同步失败**：`maybeBuildBaodanStagingSuffix` 在部分/全部拉取失败时追加 **【Agent】** 段，禁止用任意 `chat-uploads/<uuid>/` 拼**另一单**相对路径片段调 MinerU（典型 **ENOENT**）。取数**仅允许** preview 返回的**真实列名**，禁止臆造如 **`合同_合同文件`**（以 **`docs/qa_test_baodan_schema.sql`** 为准）。**勿误判未注入**：`chat-uploads/<uuid>/` 路径不含报账单号字面量，容器内 `find *TYA*` 为空属正常；以本轮合并用户消息是否含 **「报账单附件已由 BFF…」** 及 BFF 追加的 **【判据】** 段为准。**产品话术**：缺路径时**禁止**默认「请用户上传附件」；须优先引导用户写出 **TYA…** 触发 BFF，同步失败则提示**运维**配置 Next `MINIO_*` / `CHAT_UPLOAD_DIR` 等（见 `data-rule-audit-builtin-v2.md` 与 **`DATA_RULE_AUDIT_WIZARD_SYSTEM_PROMPT`**）。  
- **跨文档附件归类（合同 vs 付款通知书等）**：模型**不得**因 `影像`/`附件` 下无文件名含「付款通知书」「到账单」等关键词就停止阶段 A；应在与规则相关的**可读候选**（各目录下 PDF/图片）上 **逐项 MinerU**，依据解析出的标题、表头、正文判断哪份为合同侧、哪份为付款侧后再做字段比对。即使文件名含「合同」「付款通知书」，**仍以解析正文为准**；名实不符时须在审核原因中写明。
- **附件供 Hermes 的部署选型（对话质检、尽量少让用户操作）**：核心约束是 **MinerU/读盘工具通常只认「网关进程所在机器上」的真实路径**（或先落到该路径再解析），与「库表里的 Windows 绝对路径」不在同一文件系统时必然失败。可选路径包括：**（1）共享卷挂载（推荐私有云/同集群）**：把正式附件根（NAS、对象存储挂载点、或同步盘目录）挂进 Hermes（及 MinerU 若分容器则两边同挂）的固定路径（如 `/data/attachments`），库表仅存 **逻辑键 + 相对路径**（或 `s3://bucket/key`），部署时用环境变量拼出 **容器内绝对路径** 写入或解析，用户对话侧**零拷贝、零上传**。**（2）对象存储 + BFF 拉取（已实现 MinIO）**：Next 配置 `MINIO_*` 后，审核向导带 `baodanStageBillNo` 时 BFF **优先**从桶前缀 `MINIO_OBJECT_ROOT/<报账单号>/` 拉对象到 `chat-uploads` 并注入 Hermes 路径；可选 `BAODAN_STAGE_FALLBACK_REPO=1` 在 MinIO 失败时回退本地 `test/报账单`。**（3）HTTP 文件服务**：独立文件网关或 Next 只读路由提供带 **短期签名** 的 `GET`；若 MinerU 支持 **URL 入参**可直接拉取；否则需在 **Hermes 工具链** 中增加「下载 URL → `/tmp`」一步再调 MinerU（需网关/MCP 开发）。**（4）生产落库**：避免把开发者本机 `C:\Users\...` 写入生产库；应写 **存储键**（相对路径、对象键），由运行环境解析为可读路径。

## 对话白盒化与零干扰改造（正式版）

### 改造目标

1. 聊天界面移除昵称、头像、时间戳，保证“零干扰沉浸”。  
2. AI 回复过程拆分为“思考区 + 执行区”，并在进行中实时更新。  
3. 执行完成后自动折叠为“状态摘要条”，支持展开回看细节。  
4. 最终内容继续使用 Markdown 渲染，保留结构化结果展示能力。  

### 代码落地位置

- 消息类型扩展：`frontend/src/components/chat/types.ts`  
- 对话状态机与模拟流式：`frontend/src/store/chatStore.ts`  
- 对话渲染与折叠交互：`frontend/src/components/chat/MessageList.tsx`  

### 代码实现逻辑梳理

1. 在 `ChatMessage` 中新增 `process` 字段，统一承载 `status/startedAt/endedAt/thinkingText/executionLogs`。  
2. `sendMessage` 在写入用户消息后，立即插入一个“运行中”的 assistant 占位消息，先渲染白盒流程，再填充最终答案。  
3. 思考流采用逐字写入（短间隔定时），执行流采用逐行追加（日志节奏输出），形成“可见推理 + 可见执行”。  
4. 网络请求成功后继续按分片更新 assistant 正文，实现 Markdown 内容渐进显示；同时将执行日志中的 `In Progress` 收敛为 `Done`。  
5. 请求失败时将 `process.status` 切换为 `failed` 并追加失败日志，正文输出可重试提示。  
6. 消息渲染层中：
   - 用户消息固定右对齐浅蓝背景；
   - AI 消息固定左对齐；
   - 运行中显示思考/执行两块；
   - 「运行中 N秒」由 **`chatStore.formatDuration(startedAt)`**（无 `endedAt` 时用 **`Date.now()`**）计算；因流式阶段 **`messages` 未必每秒变化**，**`MessageList`** 在存在 **`process.status === "running"`** 的 assistant 消息时用 **`useEffect` + 1s `setInterval`** 触发 **`runningDurationTick`** 重绘耗时，并与 **贴底滚动** 联动（仅当用户未主动离开底部时随过程区增高滚到底）；
   - **有 `stepLogs` 时过程区顺序**：不再把 **`thinkingText` 按句号切分** 后与 **`stepLogs` 派生的动作条按索引 zip**（预置句会抢占索引，导致叙述与工具错位）；改为对 **`traceId` 相同的连续工具事件先合并为一条**（started + completed → 最终状态一条），再 **严格按 `stepLogs` 顺序纵向排列**；叙述句与胶囊同列展示，并对「我调用了…，我开始调用工具…」类重复前缀做 **`compactRedundantToolNarrative`** 压缩。无结构化 **`stepLogs`** 时仍走 **`executionLogs`** 与叙述的兼容布局，多余动作条改为 **`flex-col` 纵向**，避免横向一排「进行中 / 完成」误导时序。
   - 完成后显示可点击状态摘要条（成功/失败 + 耗时 + 展开箭头），展开时展示完整过程。  
7. 保留 `AdaptiveCardRenderer` 作为补充展示，仅在无正文时渲染卡片，避免视觉噪音。  
8. 状态摘要条渲染顺序位于 AI 正文上方，收起态采用极简单行文本（`已完成 xx秒` + 下拉箭头），与正文形成“过程在上、结果在下”的阅读顺序。  
9. 摘要条展开后，在正文上方以抽屉形式展示完整思考区与执行日志；收起后仅保留一行状态，不遮挡正文内容。  
10. 运行中与已完成展开态共用同一套过程区样式组件，统一为“浅紫思考块 + 深色执行日志块”，保证视觉格式一致。  
11. 过程披露区进一步改为“自然叙述”风格：去除“思考过程/执行日志”显式标题与技术块容器，改为段落化口语过程文本。  
12. 工具执行信息改为轻量动作条（灰色胶囊）内联展示，形成“像人一样讲解 + 关键动作可见”的交互样式。  
13. 首轮会话（第一条为用户输入）时，消息流容器在标题栏下方保留 `5` 个单位的顶部留白，保证标题与首条内容的垂直间距一致。  
14. 消息流底部固定保留 `10` 个单位留白，避免最后一条消息与输入区贴边，提升阅读与点击舒适度。  
15. 为避免特殊首屏场景下顶部留白被内容流覆盖，首条消息容器额外增加 `mt-5` 外边距，确保分隔线下方始终可见留白。  
16. 状态摘要条（`已完成 xx秒`）与展开后的过程内容之间增加 `5` 个单位垂直留白，提升层次分隔与可读性。  
17. 过程披露区按“自然段落 + 动作条穿插”渲染：每段叙述后可跟随对应动作胶囊，避免纯日志堆叠；同时过滤仅标点行（如独立“。”）以保持版面整洁。  
18. 对话过程展示改为“真实阶段驱动”：前端不再生成固定 `parse_intent/plan_tools` 模拟日志，而是直接消费 `/api/chat/turn` 返回的 `stepLogs`（优先 Hermes 原生阶段，其次从 phase 字段归一化），映射为可见的思考/计划/执行/反思过程。  
19. 当后端仅返回通用结果而无阶段细节时，接口层回填用户友好的四阶段文案（理解/计划/执行/反思），禁止暴露“链路路由/原始回答”等技术提示语给终端用户。  
20. 过程动作条增强可解释与可追溯性：步骤标签统一中文化（理解目标/制定计划/执行任务/结果复核），并支持点击查看该步骤的详细说明，避免仅显示 `understand() Done` 这类低可读信息。  
21. `/api/chat/turn` 支持 SSE 事件流返回（`meta/step/text/done/error`）；前端 `chatStore` 改为流式消费并实时更新过程区与正文，过程展示不再依赖“请求结束后一次性回放”。

25b1. 回合结束且已配置**直连模型**（`modelConfigs[0]`）时，**凡工具类步骤**（**`stepLogs` / 时间线** 中 **`kind === "tool"`** 或带 **`toolName`**，且非 **`kind === "phase"`**）一律批量调用 **`POST /api/chat/process-steps-summarize`**；**每批最多 12 条**（`MessageList` 内循环分片），模型返回 JSON 映射 **`id` → 一句话摘要**，写入 **`llmStepTitles`** 覆盖胶囊首行；未配置直连模型或某批请求失败则该批对应 key 从去重 ref 移除以便重试，已成功的批次保留摘要。不再用启发式白名单 **`prefersLlmStepSummary`** 跳过 LLM。  
25c. **助手消息一键复制**：在每条 **`role === "assistant"`** 消息条顶操作区右侧（与过程区上方 **`renderAssistantProcessChrome`** 等布局同一行语义，无 **`process`** 时右对齐）提供 **「复制」** 按钮；点击后 **`buildAssistantMessageCopyText`** 组装 **【状态】**（若有 **`process`**）、**【过程】**（与界面同源：**`buildProcessPlainTextForCopy`** 遍历 **`stepLogs` 合并项**或 **`thinkingText` + `executionLogs`**，每步 **`formatProcessDetailForCopy`** 输出完整详情、输入摘要、输出摘要（经 **`prettifyJsonBlock`** 排版）与追踪 ID）、**【正文】**、**【卡片】**；**`navigator.clipboard.writeText`** 失败时降级 **`textarea` + `document.execCommand("copy")`**；成功约 **2 秒** 内按钮显示 **「已复制」** 与对勾图标（**`copyFlashId`**）。  
25d. **条上运行态与过程折叠（WuKong）**：**无**消息区外全局顶栏；**时间序最后一条助手**由 **`renderAssistantProcessChrome`** 在过程区上方表达：**`waiting_first_chunk`/`sending`** 且 **`status===running`** 为灰点 +「思考中」（**`animate-pulse`**）；首包后为「运行中 + **`formatDuration`**」；**`completed`/`failed`** 为「已完成/失败 + 最终耗时」+ **Chevron**，过程详情包在 **`<details>`** 默认收起。更早助手消息过程区仍整段展开、无条上状态行。**`chatStore`** 仍按 **`stepLogs`/`text.delta`** 推进 **`process.phase`**（`streaming` 等），供上述 UI 读取。  
26. 流式链路增加“模型配置直连兜底”：当 Hermes 原生 SSE 不可用且请求携带有效 `modelConfig` 时，自动切换到系统已配置模型执行，并继续以 SSE 事件回传（含明确切换提示），避免因 Hermes provider 未配置导致对话完全不可用。  
27. 对 Hermes 运行时失败场景补充二次兜底：若已接入 Hermes 流但收到 `run.failed` 且原因为 `No inference provider configured`，服务端在同一请求内自动切换到系统模型直连并继续回传流式事件，避免“已连通 Hermes 但因 provider 未配置而直接终止”。  
28. 移除模板化四阶段兜底文案（理解/计划/执行/复核）；无 Hermes 原生中间事件时不再伪造过程链，仅显示“直连模型模式（无 Hermes 中间步骤）”单条提示，确保过程展示真实性。  
29. Hermes 流式接入顺序优化为优先 `v1/chat/completions` 原生流，其次才回退到 `v1/runs + events`，用于规避 runs 模式下“先启动 run 再订阅 events”导致的中间事件丢失竞态，提升“每次都走 Hermes”稳定性。  
30. 过程可读性改造：在 `chatStore.buildProcessFromStepLogs` 中新增“低信息日志过滤 + 语义重写”策略，自动过滤 `已获得阶段思考信息/阶段更新` 等噪声文案，并将 `tool.started/tool.completed` 转写为可读叙述（例如“我调用了某工具，正在处理/已返回结果”），保证过程披露更接近人类解释。  
31. Hermes 事件映射增强：`route.ts` 在处理 `reasoning.available`、`tool.started`、`tool.completed` 时优先消费 `summary/reasoning/detail/preview/result_preview` 等高信息字段，减少固定模板句输出，提升过程文本信息密度。  
32. Hermes 原生事件扩展接入：BFF 额外支持 `hermes.tool.progress` 与 `response.*`（`response.output_text.delta`、`response.output_item.added`、`response.completed`、`response.failed`）事件映射，统一转为前端契约后再渲染，确保过程披露基于真实执行事件而非前端模拟。  
33. 过程叙述升级为“理解->执行->结果/异常”三段式：由 `stepLogs` 中真实 `phase/tool/status` 组合生成，不依赖静态模板，提升阅读连贯性与可解释性。  

## 对话事件驱动优化（悟空流落地）

### 代码实现逻辑梳理

1. 对话事件统一为 `meta / text.delta / reasoning.delta / tool.started / tool.completed / step / turn.completed / turn.failed`，由 `frontend/src/app/api/chat/turn/route.ts` 输出。  
1a. **`streamFromHermesUpstream`** 对 OpenAI 兼容 **`chat.completion.chunk`**：在 **`choices[0].delta`** 上除 **`content` 字符串**外，解析 **多模态 `content` 数组**、**`reasoning_content` / `reasoning`**；对 **`delta.tool_calls`** 在首次出现 **`function.name`** 时去重下发 **`tool.started`**，避免模型长时间只跑工具时界面卡在「已连接 Hermes…」却无过程条。上游读循环包 **`try/catch`**，异常时下发 **`turn.failed`**（如 **`Hermes 流读取异常：terminated`**）。**长连接/超时（可配）**：**`frontend/src/lib/hermes-stream-timeout.ts`** + **`frontend/.env.local`** — **`HERMES_STREAM_CONNECT_TIMEOUT_MS`**（默认 5min，仅建连）、**`HERMES_STREAM_IDLE_TIMEOUT_MS`**（默认 60min 无字节才断）、**`HERMES_STREAM_MAX_DURATION_MS`**（默认 2h 总时长）、**`CHAT_TURN_MAX_DURATION_SEC`**（Next **`maxDuration`**，默认 7200）。Hermes 网关 **`HERMES_SSE_KEEPALIVE_SECONDS`**（compose 默认 **15s** 发 `: keepalive`，原 30s）、**`HERMES_RUN_SSE_POLL_SECONDS`**（默认 60）。修改网关 Python 后须 **`docker compose … up -d --build`** 重建 **gateway**；改 Next 后 **`npm run dev`** 重启。  
1b. **`chatOrchestrator`** 对单行 **`data:`** 做 **`JSON.parse` try/catch**，避免非 JSON 心跳行打断整轮 SSE。  
1c. **`chatStore`** 在 **`text.delta`** 时同步 **`assistant.text = accumulatedText`**，并把 **`process.thinkingText`** 在已有正文时改为「助手正在生成回复…」，避免流式阶段主区空白却仍显示「等待阶段与回复」。  
1d. **`streamChatTurn`**（**`chatOrchestrator.ts`**）在消费完 **`fetch` ReadableStream** 后返回 **`{ sawTurnCompleted }`**：仅当解析到 **`event: turn.completed`**（或兼容 **`done`**）时为 **`true`**。若连接被网关/代理超时、标签页休眠、Next dev 热更新或浏览器崩溃等**提前关闭**而未收到 **`turn.completed`**，历史上 **`chatStore` 仍会把该轮标为已完成**，表现为「最后一条工具后无终稿、像卡住」。现改为：**`sawTurnCompleted === false`** 时将该助手回合标为 **`failed`**，正文追加 **「连接提前结束」** 说明，**`runtimeInfo.retryableError`** 置真，引导用户用条上 **「重新生成」** 续写；**`sawTurnCompleted === true`** 时维持原 **「已完成」** 收尾逻辑。  
2. 事件统一携带 `seq/source/traceId/retryableError` 扩展字段，前端可直接用于时序显示、链路诊断与重试提示。  
3. 新增编排层 `frontend/src/store/chatOrchestrator.ts`，负责 SSE 读取与事件解析；**`POST /api/chat/turn` 使用 `credentials: "include"`** 以与会话 Cookie 一致；浏览器层 **`Failed to fetch` / `TypeError`** 时抛出带中文排查要点的 **`Error` 文案**，流中途断连同样包装提示，便于区分「HTTP 业务错」与「TCP/代理/dev 进程」问题。`chatStore` 仅负责状态更新与渲染数据组织。  
4. `chatStore.sendMessage` 引入轮次状态机：`sending -> waiting_first_chunk -> streaming -> completed|failed`，并写入 `process.phase`。  
5. 过程区数据全部由真实事件推导：`reasoning/tool/step` 进入 `stepLogs`，再由 `buildProcessFromStepLogs` 生成可读的思考文本与执行日志。  
5b. **界面卡在「思考中 / 已连接 Hermes，等待阶段与回复」**（已缓解）：根因多为 **Hermes 长任务（MinerU、多轮 MCP）期间上游 SSE 长时间无业务事件**，BFF 已连上并下发 **`meta`**，但前端在收到 **`text.delta` / `tool.*` / `reasoning.delta`** 前 **`process.phase` 仍为 `waiting_first_chunk`**，顶栏显示「思考中」。Hermes 网关 keepalive 仅维持 BFF↔Gateway，**不会自动变成前端事件**。缓解：**`turn/route.ts`** 按 **`HERMES_SSE_CLIENT_HEARTBEAT_MS`**（默认 20s）下发 **`turn.heartbeat`**；**`chatStore`** 收到心跳或工具事件后切 **`streaming`** 并更新 **`thinkingText`**；90s 无事件时前端提示可重试；**`MessageList`** 在本条已有工具/正文时不再显示「思考中」。若仍长时间无 **`turn.completed`**，查 Gateway 日志、MinerU、模型 400 等 **Hermes 侧**问题。  
5a. **工具胶囊长期「进行中」**（已修）：`processTimeline` 展平时，`text.delta` 插入的 **`narrative`** 会 flush 工具缓冲，旧逻辑只合并**相邻**同 **`traceId`** 的 step，易出现上方仍显示「我开始调用工具…正在准备参数」流光、下方已完成。现由 **`frontend/src/lib/process-timeline.ts`** 的 **`mergeToolStepLogs` / `consolidateFlatTimelineToolSteps` / `dedupeToolStepsByTraceId`**（按 **`traceId` 全局合并，保留首次出现位置）** 与 **`turn.completed` 时 `collapseProcessTimelineToolSteps`** 修正；BFF **`turn/route.ts`** 对 **`function_call_output`** 用 **`call_id`** 回填人类可读 **`toolName`**，**`hermes.tool.progress`** 补 **`traceId`** 并跳过同 **`traceId` 已 running** 的重复 **`tool.started`**。  
6. 错误重试从 `window` 事件改为 store action（`retryLatestTurn`），组件之间不再通过全局自定义事件通信。  
7. 新增链路可观测状态：`connectionStatus`、`runtimeInfo(upstreamMode/traceId/firstTokenLatencyMs/lastError)`，在消息区顶部实时显示。  

### 验收与回归

1. 同一轮对话内必须出现“过程在上、结果在下”渲染顺序；**最后一条助手**在条内用 **`renderAssistantProcessChrome`** 表达思考中/运行中/已完成（无消息区外独立顶栏）；过程步骤卡片默认仅摘要、技术明细在 **`<details>`** 内收起（终态过程区可整体 **`<details>`** 折叠，与 25d 一致）。  
2. Hermes 与直连模型两种上游均能产出同一事件契约，前端不再依赖上游事件命名差异。  
3. 当 Hermes provider 缺失或链路失败时，界面可见错误分类与可重试提示，不再出现“无返回但无状态解释”。

- **本地 CLI 包装**：`mcp-servers/mineru-local-mcp/` → 工具 **`mineru_local_health`**、**`mineru_local_parse`**（`child_process.execFile` 调 `MINERU_EXECUTABLE`，参数对齐上游 `mineru -p/-o/...`）。  
- **HTTP API 包装**：`mcp-servers/mineru-api-mcp/` → 工具与 MinerU FastAPI 路由对应：**`mineru_api_health`**、**`mineru_api_parse_and_wait`（推荐，大文档）**、**`mineru_api_sync_parse`**（小文件）、**`mineru_api_submit_parse_task`** / **`mineru_api_task_status`** / **`mineru_api_task_result`**；上传类工具读本地文件 multipart；可选 **`MINERU_API_ALLOWED_READ_PREFIX`**、**`MINERU_API_DEFAULT_BACKEND`**。Hermes 侧合并示例见 **`docs/Hermes-mineru-mcp.example.yaml`**；与数据源等**整段 `mcp_servers`** 见 **`docs/Hermes-mcp_servers.builtin-xingyan.example.yaml`**（含 **`hermes-agent-main/docker-compose*.yml`** 挂载 **`/opt/xingyan-mcp`**）。
- **大文档解析与「任务失败」归因（实现逻辑）**：MinerU 侧 pipeline/OCR 可能因页数多、内存或容器重启而中断；Hermes 若用同步 **`POST /file_parse`** 或容器内 Python 直调且缺 multipart，会表现为 HTTP 422/连接断开，模型易误判为「Hermes 任务失败」。**对策**：质检对话中由模型调用 **`mineru_api_parse_and_wait`**——MCP 进程 **`POST /tasks` 提交** → 按 **`MINERU_POLL_INTERVAL_MS`** 轮询 **`GET /tasks/{id}`**（状态 `pending`/`processing`/`completed`/`failed`）→ **`completed`** 时 **`GET /tasks/{id}/result`** 返回正文；若提交失败、MinerU 返回 **`failed`**、轮询/拉结果网络异常或超过 **`MINERU_POLL_MAX_WAIT_MS`**，则在 **`MINERU_PARSE_MAX_RETRIES`** 次内间隔 **`MINERU_PARSE_RETRY_DELAY_MS`** 重新提交；用尽后向 Hermes 返回结构化失败（含 `attempts`/`pollLog`）。**限制**：该工具仍占用 Hermes **单次 tool 调用**时长，SSE 需配合已放宽的 BFF/Gateway 空闲超时；彻底与对话解耦需后台任务队列（未做）。**`api_server.py`** 的 MCP 优先提示已要求大文件用 **`parse_and_wait`**，禁止 Hermes 内手写 Python 调 **`/file_parse`**。**落地**：`~/.hermes/config.yaml` 的 **`xingyan-mineru-api.env`** 可配置上述轮询/重试变量；数据规则审核技能（`platform-skills/data-rule-audit-skill/SKILL.md`、`data-rule-audit-builtin-v2.md`、`data-rule-audit-wizard.ts`）默认工具名为 **`mcp_xingyan_mineru_api_mineru_api_parse_and_wait`**。部署后需 **`docker compose ... up -d --build gateway`** 并重启 Gateway 以加载 MCP **`dist/index.js`** 与 **`api_server.py`** 提示。  
- **导出合并**：`frontend/src/lib/mcp-mineru-export-merge.ts`，由 **`GET /api/system/mcp-services/export`** 在 **`mergeDatasourceMcpIntoServersIfConfigured`** 之后调用；根据 **`MINERU_LOCAL_MCP_CLI_PATH`** / **`MINERU_API_MCP_CLI_PATH`** 写入 **`mcpServers`**，响应体增加 **`mineruLocalMcpMerged`**、**`mineruApiMcpMerged`**。环境变量说明见 **`frontend/.env.example`**。  
- **运维文档**：`docs/MinerUMCP.md`。

### 逻辑说明

1. **拆分原因**：CLI 路径依赖本机完整 MinerU 与算力，超时与日志体积大；API 路径适合「解析服务已单独起容器、MCP 只做轻量 HTTP + 读共享卷文件」。  
2. **与 Next 业务解耦**：两包均不调用统一质检平台 `app/api`，仅依赖用户环境里的 `mineru` 或 `mineru-api`。  
3. **注册方式**：Hermes 侧与数据源 MCP 相同，配置 **`command: node` + `args: [绝对路径/dist/index.js]` + `env`**；也可仅配置导出合并，由运维复制 JSON。  
4. **`mineru-api-mcp` 默认解析后端**：工具参数未传 **`backend`** 且环境变量未设置 **`MINERU_API_DEFAULT_BACKEND`** 时，MCP 向 **`POST /file_parse`** 提交的表单字段 **`backend`** 默认为 **`pipeline`**，与「仅下载 pipeline 权重、**`MINERU_MODEL_SOURCE=local`**」的常见联调一致，避免误用 **hybrid-auto-engine** 触发 VLM 依赖。已安装 VLM、需 hybrid 的部署应在 Hermes **`mcp_servers.xingyan-mineru-api.env`** 或导出用的 **`frontend/.env.local`** 中设置 **`MINERU_API_DEFAULT_BACKEND=hybrid-auto-engine`**（或每次工具调用显式传 **`backend`**）。  
5. **Sidecar 模型卷与配置持久化（`docker-compose.mineru-api.sidecar.yml`）**：**`mineru-model-cache` → `/root/.cache`** 存 ModelScope 权重；**`mineru-config` → `/opt/mineru-config`** 配合 **`MINERU_TOOLS_CONFIG_JSON=/opt/mineru-config/mineru.json`** 持久化 **`models-dir`**。仅缓存 `.cache` 时 **`docker compose … --force-recreate`** 会丢 **`/root/mineru.json`**，在 **`MINERU_MODEL_SOURCE=local`** 下易出现 **`AttributeError: 'NoneType' object has no attribute 'get'`** 与 **`failed_tasks` 上涨**。首次在容器内：**`MINERU_MODEL_SOURCE=modelscope`** 重建 → **`mineru-models-download -s modelscope -m pipeline`**（需 hybrid 再 **`-m vlm`**）→ 改回 **`local`** 重建。  
5a. **Sidecar GPU**：镜像内 **`torch` 常为 CUDA 构建**（如 `+cu130`），但若 compose **未挂载 GPU** 或 **宿主机驱动与 PyTorch CUDA 版本不匹配**，容器内 **`torch.cuda.is_available()` 为 False**，pipeline 走 **CPU**（仍可解析，大 PDF 更慢）。启用 **`gpus: all`** 后须 **`docker compose … up -d --force-recreate mineru-api`** 并升级 NVIDIA 驱动 / Docker Desktop **WSL2 GPU**；RTX 4050 等 **6GB 显存** 上大 PDF 仍可能 OOM。无 GPU 环境注释 **`gpus`/`deploy`** 段。

### Windows 主机 + WSL2 部署 `mineru-api`（与本仓库联调）

- **目标拓扑**：Hermes / Next / MCP **Node 进程**在 **Windows**；**MinerU FastAPI** 在 **WSL2 Ubuntu**（`mineru-api`），减轻 Windows 上 Python/算力栈差异带来的影响。  
- **数据路径**：`mineru-api-mcp` 在 **Windows** 上 `readFile` 用户给出的 **NTFS 绝对路径**，再以 **multipart** 调用 **`POST /file_parse`**，请求体到达 WSL 内服务；**不要求** WSL 挂载同一文件路径，与「解析服务与 MCP 同机读盘」模式不同但适用于当前拆分。  
- **网络**：`MINERU_API_BASE_URL` 填 Windows 能访问到的 API 根地址；默认可试 **`http://127.0.0.1:8000`**（依赖 WSL 的 localhost 转发）；失败时让 `mineru-api` 使用 **`--host 0.0.0.0`**，并在 Windows 侧改用 **`http://<WSL_IP>:8000`**（如 `wsl hostname -I` 取首段）。**Hermes 在 Docker、MinerU 在 WSL** 时 **`host.docker.internal:8000` 常因未在 Windows 宿主机监听而 `ECONNREFUSED`**，可改用 WSL IP，或按 **`docs/MinerUDockerSidecar.md`** 将 **`mineru-api` 以 sidecar 与 gateway 同 `hermes-net` 启动**，MCP 使用 **`http://mineru-api:8000`**；sidecar 镜像由 **`hermes-agent-main/docker/mineru-api-runtime/Dockerfile`** 构建（**`mineru[pipeline]`** + 国内 PyPI 默认镜像），**`docker compose ... --profile mineru up -d`** 与 **`MINERU_MODEL_SOURCE`**（首次可用 **`modelscope`**）见该文档。  
- **平台配置**：在 **`frontend/.env.local`** 设置 **`MINERU_API_MCP_CLI_PATH`**（指向本仓库 **`mcp-servers/mineru-api-mcp/dist/index.js`** 的 Windows 绝对路径）与 **`MINERU_API_BASE_URL`**；执行 **`npm run build`** 于该 MCP 包后重启 Next；在系统管理 → MCP 服务使用 **`GET /api/system/mcp-services/export`**，响应中 **`mineruApiMcpMerged: true`** 表示已并入 **`xingyan-mineru-api`**。  
- **上游安装**：按 [MinerU 官方仓库](https://github.com/opendatalab/MinerU) 在 WSL 内 **`uv pip install -U "mineru[all]"`**（或源码 editable），执行 **`mineru-api`** 启动；产品与功能介绍可参考 [MinerU 提取器](https://mineru.net/OpenSourceTools/Extractor)。**国内网络**下建议 **`pip`/`uv`** 使用清华等 PyPI 镜像（如 **`https://pypi.tuna.tsinghua.edu.cn/simple`**，见 **`docs/MinerUMCP.md`** §6）。**`mineru[all]`** 体积极大；低配或仅需 pipeline 时可按官方文档换用更轻 **extras**。虚拟环境宜建在 **WSL 默认用户** 的 **`~/mineru-venv`**；若装在 **`root`** 的 **`/root/mineru-venv`**，需保证启动用户能执行该路径（见 **`docs/MinerUMCP.md`** 安装用户说明）。**模型与 `/health`**：在 **`mineru-api` 同一用户**下执行 **`scripts/wsl-download-mineru-models.sh`**（内部 **`mineru-models-download -s modelscope -m pipeline`** 与 **`-m vlm`**，含图/hybrid 需 VLM），生成 **`~/mineru.json`**；**`wsl-start-mineru-api.sh`** 在**存在**该文件时默认 **`MINERU_MODEL_SOURCE=local`**，**不存在**时默认 **`modelscope`** 以免无缓存仍设 local 导致 Hub 拉取失败、**`failed_tasks` 飙升**（详见 **`docs/MinerUMCP.md`** §7.1）。  
- **许可证**：MinerU 使用 **MinerU Open Source License**；商业使用前应经法务评审。运维细节见 **`docs/MinerUMCP.md`** §6、§7；**模型下载**可在 Windows 执行 **`scripts/Run-DownloadMineruModels-Wsl.ps1 -Distro Ubuntu-24.04`**（与启动脚本相同，经 Base64 落 `/tmp` 再执行 **`wsl-download-mineru-models.sh`**，写入 **WSL 当前登录用户**的 **`$HOME/mineru.json`**）；WSL 安装完成后可用 **`scripts/Start-MineruApi-Wsl.ps1 -Distro Ubuntu-24.04`** 启动 API（若默认 WSL 非 Ubuntu 须带 **`-Distro`**）。

### 1. 日志中心（系统管理）

- **登录日志**：`POST /api/auth/login` 成功/失败时写审计表（账号、IP、UA、方式 `local|4a`）。
- **操作日志**：各写接口（access-control、datasource、skill、task、annotation）经统一 **`auditLog(action, resource)`** 中间件落库；摘要脱敏。
- **对话日志**：会话创建/消息落库时写会话索引行；详情走现有会话 API 只读回放；**不**存 Hermes tool 逐步日志。
- **权限**：`access` 矩阵新增 `sys-log-center` 及子按钮；审计角色可看全量。

### 2. 任务中心 — 测试任务

**对话触发（BFF + Agent 意图）**

1. 用户试跑样例满意后，输入「批量测试」「创建测试任务」等 → **`turn` 编排**识别 `intent.create_test_skill`。
2. 服务端拉取 **当前 `conversationId` 全量消息**，调用分析服务生成 **测试版 Skill**（业务目标/路径/经验/规则 → 写入 `SKILL.md` + `skill_versions` 表，`kind=test`）。
3. 用户确认卡（前端 **`TestSkillConfirmCard`**）→ `POST /api/skills` 或版本 API。
4. 口令「跑 100 条 / 5月12日」→ 解析为 `TaskCreatePayload`（`skillVersionId`, `filter`, `limit`）→ `POST /api/tasks` **`type=test`** → 任务 Worker 入队 **自动执行**。

**执行与展示**

- 任务表 + **item 子表**（`pending|running|succeeded|failed`）；Worker 按 Skill 调质检流水线（与对话规则审核共用 MCP/DB）。
- 列表页 Tab：**测试 | 正式**；详情：进度、明细表、**重试** `POST /api/tasks/:id/items/:itemId/retry`。
- 单条详情：基本信息 + 规则结果 + **人工标注** `POST .../annotation`（整体/规则级，不通过必填原因）。

**AI 分析（与技能自我迭代共用服务）**

- 任务 `completed` 后 **`POST /api/tasks/:id/analyze`**，body 含 `dataScope`（全量/已成功/已标注/排除失败/自定义 id 列表）。
- 服务逐 item 复盘 → 返回 `SkillOptimizationSuggestion[]` → 弹窗展示；**一键应用** → `POST /api/skills/:id/versions` 新版本并写 **skill_update_log**。

### 3. 任务中心 — 正式任务

- 测试任务详情 **「转正式」** → 向导表单（带出 Skill、Cron、数据源、并发）→ `POST /api/tasks` **`type=production`**，`parentTaskId` 溯源。
- 调度：平台 Cron 服务或 Hermes cron 触发批次；明细支持 **按执行批次/日期** 筛选。
- **AI 分析**同测试任务，数据范围增加「指定日期批次」「勾选 item」。

### 4. 任务执行日志

- 表 **`task_run_logs`**（`taskId`, `level`, `event`, `payload`, `traceId`, `operator`）。
- 任务创建/调度/item 状态变更/重试/分析/Skill 版本变更均 **`appendTaskLog`**；详情页 Tab 分页查询。

### 5. 技能中心增强

- **版本**：每次编辑或 AI 应用 → `semver` 递增 + `skill_versions` + 可选 diff UI。
- **自我迭代**：Skill 配置表 `auto_iterate_enabled`, `data_scope[]`, `interval`, `update_mode:auto|manual`；定时或条数阈值触发 **同一 analyze 服务**；自动模式直接发版，手动模式待确认。
- **授权**：`visibility: private|pending|public`；开放需管理员审核；`org_scope[]`, `allow_clone`, `allow_download`。
- **市场评论**：`skill_reviews`（仅 public 技能）；平均分聚合。
- **内置技能**：与 PRD §7.4 对齐（创建技能、接入数据源、数据质检、优化技能）。

### 6. 非功能与集成

- **对话**：BFF 已有 `turn.heartbeat`；V1.1 补工具层/对话层重试与过程 copy 友好化（见 PRD NT-16～18）。

- **DCOOS**：对外 REST 经 DCOOS 注册；对内消费电信能力用 DCOOS SDK/签名。

## Git 仓库与推送

4. **本地初始化**（PowerShell，路径按本机调整）：
   - `git init`
   - `git add .`
   - `git commit -m "chore: 初始化统一质检平台仓库"`
   - `git branch -M main`
5. **关联远程并推送**：在 GitHub / Gitee / GitLab 创建**空仓库**（勿勾选 README 初始化），再执行 `git remote add origin <仓库URL>`、`git push -u origin main`。克隆方：`cd frontend && npm install`，复制 `.env.example` → `.env.local` 后 `npm run dev`。
