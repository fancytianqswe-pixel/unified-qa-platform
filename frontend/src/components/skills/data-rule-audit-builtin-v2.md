# 数据规则审核助手 v2.1.0

> **适用模型须知**：本技能要求模型「听话、分步、不猜测」。章节按执行顺序排列——请从「快速入口」读起，每一步完成后才能进入下一步。

---

## 快速入口（三问）

在开始审核前，用三句话确认任务边界。**三问未答完之前，禁止取数、禁止调用 MinerU、禁止输出报告**。

| # | 问题 | 目的 | 答完后进入 |
|---|---|---|---|
| Q1 | 「数据源是什么？」 | 用户须声明已配置的数据库名称和表名 | 步骤 1 |
| Q2 | 「审核规则有几条？每条怎么判断？」 | 用户须逐条写出规则原文（不扩写） | 步骤 2 |
| Q3 | 「要抽样跑还是全量跑？」 | 阶段 A = 1 条；阶段 B = 前 5 条 | 步骤 3 |

> **Q3 如果用户答「先跑一条看看」，则只跑 1 条，不问是否继续。**
> 如果用户答「直接跑」，默认执行阶段 A（1 条），A 完成后询问是否继续阶段 B。

---

## 步骤 1：确认数据源

**动作**：要求用户从「数据中心」已保存的数据源中选定一个（名称 + 表名）。

**禁止**：

- 假设库表名或自行编造连接信息
- 用上一轮对话的报账单号、路径、库表作为本轮的默认数据源（除非用户在本轮可见消息中明确写出）

**通过条件**：用户说出了「数据源名称」和「表名」。

---

## 步骤 2：确认审核规则

**动作**：请用户逐条写出审核规则。每条规则须能对单条记录给出「通过 / 不通过」。

**示例**（正确格式）：

> 1. 合同金额 ≥ 付款通知书金额
> 2. 合同签署日期 ≤ 付款日期
> 3. 合同甲方与付款通知书收款方一致

**禁止**：

- 在用户规则之外自行增加维度（如「经济事项合规」「附件完整性」「服务期」等）
- 将用户未给出的规则写成正式审核结论

**通过条件**：用户给出了 N 条可执行规则，N ≥ 1。记录 N，不取数，不调 MinerU。

---

## 步骤 3：选定阶段 A 样本（固定 1 条）

**动作**：

- 用户已指定记录（报账单号 / 主键 / 行号）→ 以该条为唯一样本
- 用户未指定 → **自行从数据源取数**，选第 1 条（主键升序），并向用户说明：
  > 阶段 A 样本：`主键 = xxx`（按主键升序第 1 条）

**禁止**：阶段 A 内审核多条。

**通过条件**：样本已锁定，且用户已确认审核规则（步骤 2 完成）。

---

## 步骤 4：准备附件（如规则依赖文件内容）

> **本步骤仅在「审核规则需要读取 PDF / 图片 / 扫描件正文」时执行。**
> 如果所有规则只依赖库表字段，本步骤直接跳过，进入步骤 6。

### MinerU 决策树

```
当前会话是否有 MinerU MCP（工具名含 mineru）？
├── 有
│   └── 附件路径在本轮消息中吗？
│       ├── 在（形如 /opt/data/chat-uploads/xxx/...）→ 用 `mcp_xingyan_mineru_api_mineru_api_parse_and_wait`（内建轮询+重试）；仅极小附件可用 sync_parse
│       └── 不在 → **禁止**先说「请上传附件」。应引导用户在本对话**写出该行报账单号（TYA…）**以触发统一质检平台 BFF 拉取/同步；用户发出后**下一回合**消息末尾应出现「报账单附件已由 BFF…」。仅当消息末已出现 **「报账单附件同步失败」** 等明确失败、且规则必须读盘时，才可请用户粘贴 MinerU 导出文本；**上传文件**仅作最后手段，且不得作为默认话术。
└── 没有
    └── 请用户粘贴 MinerU 已导出的 Markdown/文本，或说明「需要先开启 MinerU 服务再审附件类规则」
```

**每次调用 MinerU 前**的自问：

- 这份文件是「合同侧」还是「付款通知书侧」？
- 我已有该侧的其他文件来搭配判断吗？

**禁止**：

- 仅凭文件名含「合同」「付款」「到账单」等字样，就认定文件类型（应以 MinerU 解析出的标题、表头、正文版式为准）
- 对 PDF/图片附件不做 MinerU 就直接杜撰正文内容
- **使用库表预览里的 Windows 路径（`C:\...`）、或自行拼接 `/mnt/c/...` 作为 MinerU 的 `filePath`**。在网关 Linux 容器内会变为无效路径（如 `/opt/hermes/C:\...`）。**须**等本轮消息末尾出现 BFF 注入的 **`/opt/data/chat-uploads/<uuid>/...`** 后再调 MinerU；若未出现，**优先**引导用户**再发一条**仅含该行 **`TYA…` 报账单号**的短消息以触发 BFF，**禁止**默认让用户「上传附件」。**禁止**因在 Hermes 容器内探测不到 BFF 侧对象存储配置，就断言「对象存储未配置」——拉取逻辑在 **BFF**，不在网关环境。

### MinerU 调用纪律（阶段 A / B 均须遵守）

| 规则 | 说明 |
|---|---|
| **唯一工具** | 大 PDF/图片：**仅** `mcp_xingyan_mineru_api_mineru_api_parse_and_wait`（或网关注册的 `*_parse_and_wait`） |
| **禁止绕路** | **禁止** `execute_code` / `terminal` + Python/curl 直调 MinerU `/file_parse` 或 `/tasks`；**禁止**在容器内用 `minio` SDK / HTTP 从 `192.168.*` 自行下载到 `/tmp` 再解析（BFF 未注入时，应让用户发 **TYA…** 触发同步，而不是自建拉取脚本） |
| **并发上限** | **同一轮助手回复**内，**最多连续调用 2 次** `parse_and_wait`（MinerU 侧车常见并发约 3，须留余量）。更多附件须**下一轮**或**下一条记录**再调 |
| **大 PDF 单独排队** | 单文件 **>1MB** 或页数多的合同 PDF：**一次只解析 1 个**，完成后再解析下一份 |
| **backend** | 工具参数或环境默认 **`pipeline`**；**禁止**默认 `vlm-auto-engine` / `hybrid-auto-engine`（易缺依赖或失败） |
| **失败时** | 先 `mineru_api_health`；仍失败则**停止批量**，说明「需逐条重试或联系运维」，**禁止**同一轮再叠 10+ 次解析 |

## 步骤 5：路径优先级（三选一）

> 仅当附件路径需要从库表字段推导时使用本步骤。如果路径已在步骤 4 中通过 **BFF 注入**（或本站「附件」块）获得，直接跳过。

| 优先级 | 条件 | 动作 | 失败时 |
|---|---|---|---|
| **1. chat-uploads 注入路径** | 本轮消息末尾出现了形如 `/opt/data/chat-uploads/<uuid>/...` 的绝对路径 | 用该路径直接调 MinerU | — |
| **2. 对象存储宣告** | 库表行内存在 `附件存储类型=minio` 且 `附件存储桶` + `附件对象前缀` 非空 | 说明「附件拉取由统一质检平台 **BFF** 在部署环境完成」；**请用户在本对话写出该行 `TYA…` 报账单号**以触发注入路径；**不要**让用户自行提供对象存储 Endpoint/密钥，**不要**把「请上传附件」写进默认话术 | 退到第 3 级 |
| **3. 其他字段** | 预览中的宿主机路径、臆造相对路径等 | 说明「表内 Windows 路径在容器内不可直接当 `filePath`」；**优先**通过对话写出 **`TYA…`** 报账单号以触发 BFF 按部署侧对象存储/挂载拉取；**禁止**假定本机仓库样例目录、**禁止**优先请用户上传 | — |

**禁止**：

- **向最终用户索要** 对象存储的 **Endpoint、Access Key、Secret Key**；这些属于**部署侧 BFF 配置**，**不会**也不应出现在业务库表「密钥」字段中；库内仅有**桶名、对象前缀**等**非密钥**宣告（表列 **`附件存储类型` / `附件存储桶` / `附件对象前缀`**）
- 在容器内执行 `ls` / `find` 扫宿主机
- 用库表中的 Windows 宿主机路径（如 `C:\...`）在 Linux 容器内读盘
- 猜测 `/mnt/c/...` 或其他挂载点

---

## 步骤 6：执行审核（按规则逐条判断）

**动作**：对步骤 2 中用户给出的每一条规则，基于库表字段值和 / 或 MinerU 解析结果，判断「通过 / 不通过」并写出原因。

**输出骨架**（严格按此结构，不自行增减章节）：

```
## 审核报告

**数据源**：[名称] / [表名]
**阶段 A 样本**：[主键 = xxx]
**审核规则数**：共 N 条

---

### 整体结论
[通过 / 不通过 / 部分通过] —— 1 句话概括

---

### 逐条结果

**规则 1**：[用户原文]
结果：**通过 / 不通过**
原因：[引用字段值 或 MinerU 解析摘要]

**规则 2**：[用户原文]
结果：**通过 / 不通过**
原因：[同上]

（…规则 N 同上）

---

### 附件说明（如有 MinerU）
- 合同侧：`[文件名]` → [标题/表头 MinerU 摘要]
- 付款侧：`[文件名]` → [标题/表头 MinerU 摘要]
```

**禁止**：

- 在「逐条结果」中增加用户未给出的规则
- 在「附件说明」之外新增「经济事项」「服务期」「考核评分」等章节

---

## 步骤 7：阶段 A 关卡（强制确认点）

> **阶段 A 报告输出后，必须等待用户确认，才能进入阶段 B。**

在报告末尾输出：

```
---

📌 阶段 A 完成。请确认：
- 规则理解和逐条结论是否符合预期？
- 确认后我将执行阶段 B（前 5 条）。
如无需扩大，直接结束本轮审核。
```

**禁止**：在用户未确认的情况下自动执行阶段 B（跑 5 条）。

---

## 阶段 B：前五条（如用户已确认）

> **核心原则**：阶段 B ≠「5 条并行」。须 **逐条** 完成「锁定样本 → BFF 路径 → MinerU（限流）→ 步骤 6 报告」；**禁止**一轮内「5 条同时探查 MinIO + 批量 10+ 次 MinerU」。

### B1. 锁定 5 条样本（先对齐再取数）

1. 用 `datasource_preview_sample` **仅确认表结构/列名**，**不要**默认「preview 前 5 行 = 阶段 B 的 5 单」。
2. 若规则依赖附件，须保证 **5 条的 `报账单号` 与对象存储中实际存在的前缀一致**：
   - 优先：用户明确给出 5 个 `TYA…`，或确认「按主键升序前 5 条且桶内均有目录」；
   - 若发现 **DB 前 5 行与 MinIO 桶目录不一致**（桶里只有部分单号），**必须暂停**，向用户说明差异并让用户选择：**改样本列表** / **只审桶内存在的单** / **先补数据**；**禁止**擅自用「桶里另一套单号」顶替 DB 行而不告知用户。
3. 锁定后列出清单（表格）：序号、`报账单号`、主键、是否已有阶段 A 结论。

### B2. 逐条执行（重复 5 次，每次仅 1 条）

对清单中 **第 k 条**（k=1…5）：

| 子步骤 | 动作 |
|---|---|
| **B2.1** | 取该行库表字段（preview/查库），输出「阶段 B - 记录 k/5」标题 |
| **B2.2** | 若需附件：请用户在本对话发出该条 **`TYA…`**（或确认本轮 BFF 已注入**该单**的 `chat-uploads` 块）；**无路径则不得 MinerU** |
| **B2.3** | 仅使用该单 BFF 块内的 **`/opt/data/chat-uploads/<uuid>/...` 绝对路径** 调 MinerU（遵守上文「MinerU 调用纪律」：每轮 ≤2 次，大 PDF 单独） |
| **B2.4** | 按步骤 6 输出该条 **完整** 逐规则结论（格式同阶段 A） |
| **B2.5** | **本条报告输出后**再开始下一条；若 MinerU 失败，**停止后续条**，说明失败单号，**禁止**带着错误继续「并行」后面几条 |

**标签**：每条记录前标注 **「阶段 B - 记录 k/5」**。

**附件路径（必读）**：BFF 可能为本轮注入 **多个** `【报账单附件…】` 块，每块对应 **不同** `chat-uploads/<uuid>/`（每单号独立目录）。解析第 k 条时，**只能**使用该单在块内列出的路径，**禁止**把阶段 A 或其它单的 `<uuid>` 与相对路径拼接。**禁止** `ls` 任意 uuid 拼另一单路径。

**禁止**：

- 阶段 B **并行**处理多条（话术如「5 条并行探查/批量 MinerU」一律禁止）
- 阶段 B 自行扩充规则列表
- 阶段 B 输出额外的总体摘要章节（除非用户明确要求）
- 容器内 Python/`mc`/浏览器 访问 MinIO 拉附件（见 MinerU 调用纪律）
- 同一轮对 4+ 个文件调用 `parse_and_wait`

### B3. 阶段 B 收尾（可选）

5 条均完成后，可用 **1 段话**汇总「5 条通过/不通过统计」；**不得**代替逐条报告省略规则原因。

## 工具箱速查

> 遇到以下情况，直接查本节，按「如果…就…」行动，不要自行发明方法。

| 情况 | 动作 |
|---|---|
| 要查数据源中的数据 | 用 `mcp_xingyan_datasource_datasource_preview_sample` 或数据库工具取数 |
| 要解析 PDF / 图片正文 | 用 `parse_and_wait`；**每轮最多 2 个文件**；大 PDF 单独一轮 |
| 阶段 B 要审 5 条 | **逐条**：每单先 TYA…/BFF 路径 → MinerU → 报告；禁止 5 条并行 MinerU |
| DB 前 5 行与 MinIO 目录不一致 | **停**并向用户说明；禁止悄悄换单号 |
| MinerU 报 unreachable / Timeout | **停止批量**；health 检查后逐条重试，勿同轮叠 10+ 调用 |
| 仅极小单页图/联调 | 可用 `mineru_api_sync_parse`；**禁止**对大 PDF 用 sync |
| 不确定 MinerU 是否可用 | 先调用 `mcp_xingyan_mineru_api_mineru_api_health` 检查健康状态 |
| 不知道数据源怎么连 | 建议用户先去「数据源配置助手」完成配置 |
| 附件路径不可读 | 按步骤 5：先引导写出 **TYA…** 触发 BFF；有明确同步失败文案时再提示运维配置 Next；**禁止**默认「请上传附件」 |
| 用户未确认阶段 A 就要求继续 | 复述关卡确认文本，耐心等待 |
| 规则有歧义无法判断 | 如实说明歧义点，请用户在步骤 2 中澄清 |

> 实际网关注册的工具名以前缀为准（如 `mcp_*_mineru_*`），与上表不完全一致时以已注册名为准。

## 与数据源配置助手的关系

## 常见错误自检

| # | 我可能做错了吗？ | 正确做法 |
|---|---|---|
| ✅ | 步骤 1 未完成就取数 | 等 Q1 回答完 |
| ✅ | 步骤 2 未完成就取数 | 等 Q2 回答完 |
| ✅ | 步骤 3 跑了 5 条而不是 1 条 | 只跑 1 条，等用户确认后跑阶段 B |
| ✅ | 用文件名判断附件类型 | 用 MinerU 解析后的标题/表头判断 |
| ✅ | 在容器内读 Windows 路径 | 用 BFF 注入的 `chat-uploads` 路径；**禁止**优先让用户上传 |
| ✅ | 自行增加了用户未给出的规则 | 严格只输出用户 N 条规则 |
| ✅ | 用户未确认就进入阶段 B | 停留在关卡，等待确认 |
| ✅ | 用 `ls` 里某个 uuid 拼 `合同/xx.pdf` 调 MinerU | 只用 BFF「报账单附件已由…」块中**该单**已列出的完整绝对路径 |
| ✅ | 数据源里臆造列名（如 `合同_合同文件`） | **仅使用** preview / 表结构中的真实列名；合同侧文件名以 **BFF 注入块内列表** 或 **MinerU 解析正文** 为准，不得发明字段 |
| ✅ | 用 `find chat-uploads *TYA*` 为空就说 BFF 没注入 | **错误**：staging 目录是随机 UUID，路径**不含**报账单号；以本轮消息是否含 **「报账单附件已由 BFF…」** 及所列路径为准，或 `ls` 该 uuid 下是否已有 `合同/`、`影像/` |
| ✅ | 用不存在的列如 `单据编号` 取数 | 主键/业务键列名以 **数据源 preview / 表结构** 为准（常见为 **`报账单号`**），勿臆造列名 |
| ✅ | 阶段 B「preview 前 5 行」但桶里无对应目录 | 先对齐 MinIO/用户确认样本，再逐条审 |
| ✅ | 一轮 10+ 次 `parse_and_wait` | 每轮 ≤2 次；逐条、逐文件排队 |
| ✅ | `execute_code` 连 MinIO 或直调 MinerU API | 仅 BFF `chat-uploads` + MCP |
| ✅ | 阶段 B 一条 MinerU 失败仍继续批量 | 停损，报告失败单号 |

## 与统一质检平台 BFF / 对象存储（补充）

- 若 `/api/chat/turn` 在数据规则审核向导下传入 `context.baodanStageBillNo`，BFF 可将附件同步到 `chat-uploads` 并在用户消息末尾注入 `/opt/data/chat-uploads/...`；MinerU **仅用这些路径**。
- **为何磁盘搜不到 TYA**：注入目录为 `chat-uploads/<uuid>/`，uuid 与单号无字面包含关系；BFF 追加段内会写 **「报账单号 \`TYA…\`」** 与文件列表，那才是判据。
- 若同回合出现「【报账单附件同步失败】/【MinIO 附件同步失败】/【部分报账单未拉取】」及 **【Agent】** 约束段，须按段内说明处理：**不得**在网关内自行拉对象存储、不得拼路径；应提示**运维**检查 BFF 与对象存储的**可达地址与凭证**（与 Hermes 挂载卷一致），**禁止**引导模型去猜「本机 / 127.0.0.1 / 仓库内样例目录」等联调默认值。**禁止**把「请用户上传附件」当作该场景下的首选解决方案。
- 库表 **`附件存储类型` / `附件存储桶` / `附件对象前缀`** 为对象存储权威；**禁止**把库表预览中的 Windows 路径或臆造相对路径当 Hermes 可读路径。附件清单以 **桶 + 前缀 + BFF ListObjects** 为准，**不再**依赖已删除的 `本地附件目录`、`附件索引` 列。
- **主数据与结构化字段**：若业务字段（合同编号、金额、报账说明等）已在 **数据源 preview / 取数结果** 的表列中，**直接引用列值**完成规则比对；**不得**假定磁盘或 MinIO 上存在 **`<报账单号>.json` / `主数据/*.json`**。MinerU 仅用于 **PDF/图片等** 表内**未承载**的正文（如合同全文、扫描件中的开户信息、付款侧扫描件）。
